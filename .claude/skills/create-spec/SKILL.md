# /create-spec — Spec Creation Workflow

You are the lead orchestrator. You coordinate spec writing, review, and PR creation.
Never write the spec yourself — spawn agents to do the work.

---

## Step 0 — Parse arguments

`$ARGUMENTS` contains the feature name and optional `--issue N` flag.

Parse:
- `FEATURE_NAME` — kebab-case feature name (first argument, strip special chars, max 40 chars)
- `ISSUE_NUMBER` — optional GitHub issue number from `--issue N` flag

Set variables:
```
FEATURE_NAME    = {derived kebab-case name}
SPEC_DIR        = .claude/tasks/{FEATURE_NAME}/
SPEC_PATH       = .claude/tasks/{FEATURE_NAME}/spec.md
TASKS_PATH      = .claude/tasks/{FEATURE_NAME}/tasks.json
SPEC_BRANCH     = spec/{FEATURE_NAME}
```

---

## Step 1 — Gather context

If `ISSUE_NUMBER` is provided, fetch the issue:
```bash
gh issue view {ISSUE_NUMBER} --json title,body --jq '.title + "\n\n" + .body'
```

Set `FEATURE_DESC` to the issue content, or to `$ARGUMENTS` if no issue was provided.

---

## Step 2 — Write spec (spawn spec-writer agent)

1. Read `.claude/skills/create-spec/phase-write.md`
2. Spawn the **spec-writer** agent (`subagent_type: spec-writer`, model: opus) with that
   file's content as the prompt, appending:

```
--- CONTEXT ---
FEATURE_NAME:  {FEATURE_NAME}
FEATURE_DESC:  {FEATURE_DESC}
SPEC_DIR:      {SPEC_DIR}
SPEC_PATH:     {SPEC_PATH}
TASKS_PATH:    {TASKS_PATH}
```

3. Wait for the agent to finish.
4. Verify `SPEC_PATH` and `TASKS_PATH` both exist. If either is missing, re-spawn once.

---

## Step 3 — Review spec (spawn spec-reviewer agent)

1. Read `.claude/skills/create-spec/phase-review.md`
2. Spawn the **spec-reviewer** agent (`subagent_type: spec-reviewer`, model: sonnet) with
   that file's content as the prompt, appending:

```
--- CONTEXT ---
SPEC_PATH:     {SPEC_PATH}
TASKS_PATH:    {TASKS_PATH}
```

3. Wait for the reviewer to return findings.

### Handle review results

If `status: issues-found`:
- Show findings to the user
- Ask the user if they want to fix the issues
- If yes, re-spawn the spec-writer agent with the findings appended under `--- REVIEW FINDINGS ---`
- Run one more review pass after fixes
- Accept the result and proceed

If `status: pass`:
- Proceed to Step 4

---

## Step 4 — Create branch and PR

```bash
# Create spec directory if needed
mkdir -p ".claude/tasks/{FEATURE_NAME}"

# Create and switch to spec branch (handle existing branch)
git rev-parse --verify "spec/{FEATURE_NAME}" >/dev/null 2>&1 \
  && git checkout "spec/{FEATURE_NAME}" \
  || git checkout -b "spec/{FEATURE_NAME}"

# Stage spec files
git add ".claude/tasks/{FEATURE_NAME}/"

# Commit
git commit -m "spec({FEATURE_NAME}): add feature spec and tasks"

# Push
git push -u origin "spec/{FEATURE_NAME}"

# Create PR
gh pr create \
  --title "spec({FEATURE_NAME}): feature specification" \
  --body "$(cat <<'EOF'
## Feature Spec

Spec for **{FEATURE_NAME}** ready for review.

### Files
- `.claude/tasks/{FEATURE_NAME}/spec.md` — full feature specification
- `.claude/tasks/{FEATURE_NAME}/tasks.json` — machine-readable task list

### Tasks
**Backend:**
{list B tasks}

**Frontend:**
{list F tasks}

### Next steps
1. Review this spec PR
2. Merge to main when approved
3. Run `/implement {FEATURE_NAME}` to start implementation

Generated by the /create-spec workflow.
EOF
)" \
  --base main \
  --head "spec/{FEATURE_NAME}"
```

Return to main branch:
```bash
git checkout main
```

---

## Step 5 — Report

Output the PR URL so the user can review the spec.

---

## Global rules

- Never write the spec yourself — always use the spec-writer agent
- The spec-writer agent works interactively with the user (presents options, waits for choices)
- Keep a running status log: `[Step 0 done] [Step 1 done] [Step 2 running...] ...`
- If a step fails, retry once before reporting the error
