## Codebase Patterns
- Domain folder structure: `models/` (interfaces), `pipes/` (Angular pipes), `util/` (pure functions only), `feature/{name}/` (one subfolder per smart component), `{domain}.routes.ts` at domain root
- Global styles: `tailwind.css` (plain CSS, `@import 'tailwindcss'`, PostCSS only) + `styles.scss` (SCSS-specific styles). Never merge these two files.
- Sheriff `layer:pipes` dep rule: `'layer:pipes': ['layer:util']` — pipes can import from util. Also update `layer:feature` and `layer:ui` to include `layer:pipes`.
- Sheriff's `<layer>` tagging pattern auto-creates `layer:<foldername>` tags for each subfolder — any new folder (e.g. `models/`) needs a `layer:models` dep rule added to sheriff.config.ts
- Templates: always use `templateUrl:` with a separate `.html` file — inline `template:` is forbidden by ESLint
- CI: `nx affected` for PRs must use `--base=origin/main` (not bare `main`) — add `git fetch origin main` step first
- After creating PR: run `gh run watch --exit-status` to wait for CI before signaling COMPLETE
- Playwright e2e: `getByText('X')` partially matches — use `getByRole('heading', { name: 'X' })` for heading assertions to avoid strict mode violations
- Playwright e2e: Use `page.keyboard.press('Escape')` to dismiss native `showModal()` dialogs cross-browser; clicking inside dialogs is unreliable in Firefox/WebKit with Angular 21 zoneless mode
- Interface re-exports need `export type { }` when `isolatedModules: true`
- Sheriff needs explicit dep rules for: 'root' (app files) → ['chairly-lib', 'shared']; library barrel tags for each lib
- Form label elements MUST have `for="id"` matching `id="..."` on inputs — enforced by angular-eslint template rules
- For `<input type="number">` tests in jsdom: use `new Event('input')` not `new Event('change')` to update form control values
- `.gitkeep` files: remove as soon as a folder has real content

## Codebase Patterns
- Sheriff requires an `index.ts` barrel in each tagged directory for proper module tagging (e.g., `data-access/index.ts`).
- Sheriff's tagging for `libs/shared/src` must be added as a direct entry (not under `libs/shared/src/lib`) so that barrel imports (`@org/shared-lib`) resolve to a tagged module.
- Sheriff uses AND logic for multi-tag dep rules — if a file has tags `[domain:X, layer:data-access]`, BOTH rules must allow the dependency. So `layer:data-access` must include `'shared'` in its allowed deps for data-access services to import from shared-lib.
- For `Observable<void>` delete methods in tests, flush with `req.flush(null)` and verify completion with a boolean flag (not `toBeUndefined()` — HttpClient returns null not undefined for empty bodies).
- Interface re-exports in barrel files need `export type { }` syntax when Angular build uses `isolatedModules: true`. Use `export type { MyInterface }` instead of `export { MyInterface }` for all TypeScript interfaces.
- Sheriff needs 'root' dep rules for app (apps/ directory) files, plus a 'chairly-lib' tag for `libs/chairly/src` barrel. Without these, Sheriff raises SH-002 errors.
- Smart container test pattern: mock stores as `{ signal(), vi.fn() }` plain objects, provide via TestBed, use `triggerEventHandler('eventName', payload)` to test event wiring.
- All form `<label>` elements MUST have `for="id"` attributes matching `id="..."` on the associated control — `@angular-eslint/template/label-has-associated-control` enforces this.
- For `<input type="number">` in Angular tests (jsdom): dispatch `input` events, NOT `change` events — `change` events don't reliably update the form control value in jsdom. Use `new Event('input')` instead.
- The git working directory is /home/jreuvers/projects/Chairly but `pwd` inside frontend tools resolves to /home/jreuvers/projects/Chairly/src/frontend/chairly. Use `git -C /home/jreuvers/projects/Chairly` for git commands.
- Frontend Nx quality checks are run from /home/jreuvers/projects/Chairly/src/frontend/chairly: `npx nx affected -t lint --base=main`, etc.
- npm install must be run once in src/frontend/chairly before Nx commands work (node_modules not committed).
- import-sort rule uses `simple-import-sort/imports` — Angular imports are sorted alphabetically by package name (@angular/common/http before @angular/core).
- SonarJS `no-nested-functions` enforces max 4 levels of function nesting — extract map/filter callbacks used inside patchState (which is inside subscribe which is inside withMethods) to module-level helper functions.
- signalStore() creates an injectable class — provide in TestBed; mocked dependencies via { provide: X, useValue: mockObj }; of() is synchronous so no fakeAsync needed.
- eslint --fix auto-sorts imports/exports — run it after creating files to avoid manual sort guessing.
- `@angular-eslint/template/no-call-expression` does NOT support Angular signal reads — it blocks all function calls including `signal()` reads. It has been disabled in eslint.config.mjs (rule source confirmed no TypeScript type checking). Signal reads in templates remain the approved Angular 17+ pattern.
- Angular's HTML parser (used by @angular-eslint) treats `<dialog>` as having implied end tags in some versions — the `Parsing error: '>' expected` at line 18:6 was a misleading error from a `</dialog>` closing tag mismatch (opening tag was accidentally changed to `<div>` but closing tag left as `</dialog>`). Always use matching `<dialog>...</dialog>` tags.
- Two separate `import ... from '@angular/core'` statements cause `simple-import-sort/imports` errors and `Parsing error: '>' expected` in the inline template processor. Always merge all Angular core imports into one statement.
- Angular signal inputs (`input()`, `input.required()`) and outputs (`output()`) must be marked `readonly` — `@angular-eslint/prefer-signals` enforces this. Run `eslint --fix` to auto-fix.
- For testing Angular components with native `<dialog>` element: jsdom doesn't implement `showModal()` or `close()`. Assign mock functions directly: `dialogEl.showModal = vi.fn(); dialogEl.close = vi.fn();` after `fixture.detectChanges()`.
- Signal outputs (`output<T>()` returns `OutputEmitterRef<T>`) have a `.subscribe()` method usable in tests: `component.output.subscribe(() => { done = true; })`.
- `viewChild.required<ElementRef<HTMLDialogElement>>('name')` requires the template reference variable name to match the string parameter. Use a non-conflicting name like `#dialogEl` (not `#dialog` which may cause issues).
